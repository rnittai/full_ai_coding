# ブランチ粒度・実装工程作成フェーズ指示書

最終的な目標を達成するために、ブランチの粒度のタスクに分解します。このフェーズでは、まだ実装を開始してはいけません。

---

## 目的

実装作業を安全に進められるように、ブランチ単位の作業計画を作成すること。

このフェーズの成果物は、以下のファイルとします。

```text
./ai/goal_branch_name.txt
./ai/implementation_branch_plan.md
```

---

## デフォルトで許可された操作

以下の操作は、追加確認なしで実施してよいものとします。

- ファイルの読み取り
- git の read only な操作
  - 例:
    - `git status`
    - `git branch`
    - `git log`
    - `git diff`
    - `git show`
    - `git remote -v`
- 必要なインターネット検索
- 以下のファイルの作成・編集
  - `./ai/goal_branch_name.txt`
  - `./ai/implementation_branch_plan.md`

---

## デフォルトで禁止された操作

明示的な許可がない限り、以下は行ってはいけません。
-  `./ai/goal_branch_name.txt` と `./ai/implementation_branch_plan.md` 以外のファイル作成・編集
- ファイルの削除・移動・リネーム
- 依存関係の追加・削除・更新
- パッケージインストール
- テスト・ビルド・フォーマット・lint など、リポジトリに副作用を与える可能性のあるコマンド
- git の write 操作

# 実施手順

## 1.  `goal branch` の名前の決定

現在のブランチから切り、最終的に今回の実装成果を統合する先として想定する `goal branch` の名前を決定してください。

`goal branch` は、実装作業全体の到達点を表す計画上のブランチ名です。
`goal branch` 名は、以下の観点で決めてください。

- ユーザー要望全体を表していること
- 個別実装ブランチよりも上位のまとまりを表していること
- 既存のブランチ命名規則がある場合はそれに従うこと
- 既存ルールが不明な場合は、`goal/<short-purpose>` または `feature/<short-purpose>` を推奨すること
- 実装ブランチが複数ある場合でも、それらを最終的に集約する対象として理解しやすいこと

決定した `goal branch` 名は、 `./ai/goal_branch_name.txt` に記載してください。当該ファイルにはブランチ名のみをプレーンテキストで保存してください。

## 2. ブランチ粒度の設計

ユーザーの要望を実現するために、`goal branch` から切るブランチの粒度のタスクに分割してください。

ブランチ粒度は、以下の基準で決めてください。

- 1ブランチにつき、目的が1つにまとまっていること
- レビューしやすい変更量であること
- 他ブランチへの依存関係が明確であること
- 途中で中断してもリポジトリが壊れにくいこと
- 可能であれば、小さい順に安全にマージできること
- テスト・検証単位が明確であること
- リファクタリングと機能追加を可能な限り分離すること
- 大規模変更の場合は、基盤整備・実装・検証・移行を分けること

---

以下のファイルを作成または更新して、「3. ブランチ粒度の設計」で作成したタスクをまとめてください。

# ブランチ名の方針

ブランチ名は、リポジトリの既存ルールが確認できる場合はそれに従ってください。

既存ルールが不明な場合は、以下の形式を推奨してください。

```text
feature/<short-purpose>
fix/<short-purpose>
refactor/<short-purpose>
test/<short-purpose>
docs/<short-purpose>
```

複数ブランチに分ける場合は、順序がわかるようにしてもよいです。

例:
```text
feature/01-base-setup
feature/02-domain-logic
feature/03-ui-integration
feature/04-tests-docs
```

ただし、実際にブランチを作成してはいけません。
あくまで計画として記載してください。

---

# 判断基準

ブランチを分けるべき例:

- 既存コードの整理と新機能追加が混在する
- DB・API・UIなど影響範囲が異なる
- 先に土台となる設計変更が必要
- テスト追加を別工程にした方がレビューしやすい
- リスクが高い変更を独立させたい

ブランチを分けすぎない方がよい例:

- 1つの小さな修正で完結する
- ファイル数・差分量が少ない
- 分割すると依存関係がかえって複雑になる
- 1つのテストでまとめて確認する方が自然
