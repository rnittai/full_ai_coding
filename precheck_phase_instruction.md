# 事前確認フェーズ指示書

あなたは、ユーザーから実現したい要望を受け取った後、すぐに実装を開始してはいけません。

まず、以下の順序で事前確認を行ってください。

## 目的

ユーザーの要望を安全かつ適切に実現するために、現在のリポジトリ状況を確認し、必要な情報を調査し、人間に対して事前確認・質問・助言を `./ai/precheck_questions_advice.md` にまとめること。

## デフォルトで許可された操作

以下の操作は、追加確認なしで実施してよいものとします。

- ファイルの読み取り
- git の read only な操作
  - 例:
    - `git status`
    - `git branch`
    - `git log`
    - `git diff`
    - `git show`
    - `git remote -v`
- 必要なインターネット検索
- `./ai/precheck_questions_advice.md` の作成・編集

## デフォルトで禁止された操作

明示的な許可がない限り、以下は行ってはいけません。
- `./ai/precheck_questions_advice.md` 以外のファイル作成・編集
- ファイルの削除・移動・リネーム
- 依存関係の追加・削除・更新
- パッケージインストール
- ビルド・テスト・フォーマット・lint など、リポジトリに副作用を与える可能性のあるコマンド
- git の write 操作
  - 例:
    - `git add`
    - `git commit`
    - `git checkout -b`
    - `git switch -c`
    - `git merge`
    - `git rebase`
    - `git push`
    - `git reset`

# 実施手順

## 1. 現在のリポジトリ状況の確認

まず、現在のリポジトリの状態を確認してください。

確認対象には、少なくとも以下を含めます。

- 現在のブランチ
- 作業ツリーの状態
- 未コミット変更の有無
- 直近のコミット履歴
- リポジトリ構成
- 関連しそうなディレクトリ・ファイル
- README や開発ドキュメントの有無
- 使用技術・フレームワーク・言語
- テスト・ビルド・lint などの実行方法が記載されているか

ただし、この段階ではファイルの編集は行ってはいけません。

使用してよいコマンド例:

```sh
git status
git branch --show-current
git log --oneline -n 20
git diff --stat
git diff
ls
find . -maxdepth 3 -type f
```

必要に応じて、関連ファイルを読み取ってください。

---

## 2. 必要な情報の調査

ユーザーの要望を実現するために必要な情報を調査してください。

調査対象には、必要に応じて以下を含めます。

- リポジトリ内の既存実装
- 関連する設計・仕様・命名規則
- 既存のテスト
- 既存の設定
- 使用ライブラリやフレームワークの仕様
- 外部 API や外部サービスの仕様
- インターネット上の公式ドキュメントや信頼できる情報

インターネット検索を行う場合は、原則として以下を優先してください。

1. 公式ドキュメント
2. 公式 GitHub リポジトリ
3. 標準仕様
4. 信頼できる技術記事

調査結果は、後続の事前確認・質問・助言に反映してください。

---

## 3. 人間に対する事前確認・質問・助言の作成

調査が完了したら、以下のファイルを作成または更新してください。

```text
./ai/precheck_questions_advice.md
```

このファイルには、ユーザーが実装前に確認すべき内容を整理して記載してください。

必ず以下の構成にしてください。

```md
# 事前確認・質問・助言

## 1. 要望の理解

< ユーザーの要望を、現在のリポジトリ状況を踏まえて簡潔に整理する >

## 2. 現在のリポジトリ状況

< 確認した内容を記載する >

例:
- 現在のブランチ
- 作業ツリーの状態
- 関連しそうなファイル・ディレクトリ
- 既存実装の概要
- 既存テストや検証方法の有無

## 3. 調査結果

< 実装判断に必要な調査結果を記載する >

例:
- リポジトリ内で確認したこと
- 外部情報を調査した場合は、その要点
- 参照した公式ドキュメントや情報源

## 4. 実装前に人間へ確認したいこと

< 実装前に確認が必要な事項を、質問形式で記載する >

例:
- この仕様で進めてよいですか？
- 既存の挙動を変更してよいですか？
- 後方互換性を維持する必要がありますか？
- UI / API / データ構造のどれを優先すべきですか？
- エラー時の挙動はどうするべきですか？

## 5. 実装前の助言

< 人間に対する助言を記載する >

例:
- 想定されるリスク
- 影響範囲
- 実装方針の選択肢
- 推奨する進め方
- 先に決めるべきこと

## 6. 次に人間から欲しい判断

人間が次に返答しやすいように、選択肢や確認項目をまとめる。

例:
- A案で進める
- B案で進める
- 追加調査する
- 仕様を修正する
```

---

# 完了条件

以下を全て満たしたら、事前確認フェーズは完了です。

- 現在のリポジトリ状況を確認している
- 必要な情報を調査している
- `./ai/precheck_questions_advice.md` を作成または更新している
- 人間が判断しやすい質問・助言を記載している

---

この時点では、実装に進んではいけません。
人間から明示的な許可が出るまで待機してください。
