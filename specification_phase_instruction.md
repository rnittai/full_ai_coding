# 仕様確定フェーズ指示書

実装の仕様を確定します。
このフェーズでは、まだ実装を開始してはいけません。

---

## 目的

ユーザーの回答をもとに、実装で迷わない粒度まで仕様を確定し、後続のブランチ計画・実装工程作成の前提となる仕様書を作成すること。

このフェーズの成果物は、以下のファイルとします。

```text
./ai/confirmed_specification.md
```

---

## デフォルトで許可された操作

以下の操作は、追加確認なしで実施してよいものとします。

- ファイルの読み取り
- git の read only な操作
  - 例:
    - `git status`
    - `git branch`
    - `git log`
    - `git diff`
    - `git show`
    - `git remote -v`
- 必要なインターネット検索
- 以下のファイルの作成・編集
  - `./ai/confirmed_specification.md`

---

## デフォルトで禁止された操作

明示的な許可がない限り、以下は行ってはいけません。

- `./ai/confirmed_specification.md` 以外のファイル作成・編集
- ファイルの削除・移動・リネーム
- 依存関係の追加・削除・更新
- パッケージインストール
- テスト・ビルド・フォーマット・lint など、リポジトリに副作用を与える可能性のあるコマンド
- git の write 操作

---

# 実施手順

## 1. 事前確認内容とユーザー回答の読み取り

まず、以下のファイルを読み取ってください。
- `./ai/answer_for_precheck_questions.md`: `./ai/precheck_questions_advice.md` に対しての人間の回答が記載されています。

---

## 2. ユーザー回答の仕様化

これまでの内容と `./ai/answer_for_precheck_questions.md` の内容をもとに、回答を仕様として整理してください。

以下の観点で分類してください。

- 確定した仕様
- 確定した非対応事項
- 採用する実装方針
- 採用しない実装方針
- 既存挙動を維持する部分
- 既存挙動を変更する部分
- UI / API / データ構造 / 設定 / ドキュメントなどへの影響
- テストや検証で満たすべき条件
- 実装時に判断してはいけない事項
- 後続フェーズで追加確認が必要な事項

ユーザーの回答から明確に読み取れない内容を、勝手に確定仕様として扱ってはいけません。

ただし、実装計画に進めるために仮定が必要な場合は、必ず「仮仕様」または「未確定事項」として明記してください。

---

# 仕様確定時の判断ルール

## ユーザー回答を最優先する

`./ai/answer_for_precheck_questions.md` に明記された人間の回答を最優先してください。

事前確認時の助言とユーザー回答が矛盾する場合は、原則としてユーザー回答を採用します。
ただし、明らかなリスクがある場合は、仕様書の「制約・注意点」または「未確定事項」に明記してください。

---

## 勝手に仕様を追加しない

ユーザーが明示していない要件を、確定仕様として追加してはいけません。

必要に応じて、以下のいずれかに分類してください。

- 確定仕様
- 非対応
- 未確定事項
- 仮仕様
- 実装中に判断可能な事項
- 後続フェーズで検討する事項

---

## 実装者が迷わない粒度まで整理する

仕様書は、後続のブランチ計画・実装工程作成の前提になります。

そのため、以下が曖昧なままにならないようにしてください。

- 何を実装するのか
- 何を実装しないのか
- 既存挙動を変えるのか
- 既存挙動を維持するのか
- エラー時にどうするのか
- データをどう扱うのか
- UIやAPIの挙動はどうなるのか
- どの観点で検証するのか
