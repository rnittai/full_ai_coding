# 確定仕様書

## 1. 概要

### 1.1 目的

ブラウザエンジンを使用せず、Rust でネイティブな軽量 Discord 風チャットアプリを作成する。

初回実装では、同一マシン上でサーバー 1 台とクライアント 2 台を起動し、サインアップ・ログイン後にサーバー / チャンネル階層上でメッセージを送受信できる状態を目標とする。

### 1.2 背景

ユーザーは Electron / Tauri / WebView のようなブラウザエンジン依存を避け、Rust ネイティブ GUI と Rust サーバーで軽量なチャット体験を実現したい。将来的には Windows / macOS 対応も視野に入れるため、OS 固有処理と共通処理は分離しやすい構成にする必要がある。

### 1.3 対象範囲

- Rust Cargo workspace の新規構成
- 共通データ型を持つ `common` crate
- SQLite 永続化を持つチャットサーバー
- `iced` によるネイティブ GUI クライアント
- `tokio` + `axum WebSocket` + `serde_json` による通信
- サインアップ、簡易ログイン
- パスワードのソルト付きハッシュ保存
- サーバー / チャンネル階層の表示
- チャンネル作成、チャンネル削除
- チャンネル単位のテキストメッセージ送受信
- クライアント 1 とクライアント 2 のメッセージ疎通確認

### 1.4 対象外

- Discord 公式 API との連携
- Discord アカウントとの連携
- 音声通話、ビデオ通話、画面共有
- 添付ファイル、画像アップロード、絵文字、リアクション
- DM、フレンド、招待リンク
- ロール、権限、モデレーション
- Push 通知、OS ネイティブ通知
- 本番運用向け TLS、公開サーバー運用、外部認証連携
- Windows / macOS での実動作保証
- サーバー作成・削除機能。ただし、サーバー / チャンネル階層を UI とデータ構造として持つ

## 3. ユーザー回答から確定した事項

### 3.1 採用する方針

- Discord API 連携ではなく、Discord 風の独自チャットアプリとして作る
- 同一マシン上で `server`、`client1`、`client2` を起動してメッセージ交換できる MVP を作る
- クライアント UI は `iced` を使う
- サーバーは `tokio` + `axum WebSocket` を使う
- クライアント・サーバー間データは `serde_json` で扱う
- メッセージ履歴は SQLite に保存する
- ユーザー認証として簡易ログインを入れる
- サインアップを入れる
- パスワードは平文保存せず、ソルトを加えてハッシュ化したものを保存する
- Discord 風のサーバー / チャンネル階層を作る
- 新規チャンネル作成とチャンネル削除に対応する
- 対象 OS は今回の開発環境で動けばよい
- 将来の Windows / macOS 対応を見据え、OS 固有処理と共通処理を分ける
- サーバーとクライアントは同一リポジトリの Cargo workspace として作る
- アプリ本体はリポジトリ直下に作る
- 共通メッセージ型を `common` crate に分ける
- メッセージ構造には最初から `user_id`、`message_id`、`timestamp` を含める
- 実装後に `cargo check`、`cargo test`、`cargo fmt --check`、可能なら `cargo clippy -- -D warnings`、手動疎通確認を行う

### 3.2 採用しない方針

- `egui/eframe` による最短 MVP
- `Slint` による軽量ネイティブ UI
- ブラウザエンジンを使う UI 実装
- メッセージ履歴を保存しないインメモリのみの実装
- 認証なしの表示名だけの接続
- 1 つの共通チャットルームだけの実装
- `ai/` 配下にアプリ本体を作る方針

### 3.3 条件付きで採用する方針

- `cargo clippy -- -D warnings` は「可能なら」実施する。依存ライブラリや環境都合で実行できない場合は、理由を記録する。
- Windows / macOS 対応は将来のために分離設計を意識するが、今回の実動作保証対象にはしない。
- パスワードハッシュ方式は、ユーザー指定の「ソルト付きハッシュ」を満たす方式として Argon2id を仮仕様とする。実装時に技術的制約が判明した場合は、平文保存には戻さず、bcrypt / scrypt などの専用パスワードハッシュへ変更する判断を人間に確認する。

### 3.4 既存挙動を維持する事項

- 既存のアプリ挙動は存在しないため、維持対象の既存機能はない
- `ai/` 配下の作業フロー用ファイルはアプリ実装対象にしない
- `ai/.gitignore` の扱いはこの仕様では変更しない

### 3.5 既存挙動を変更する事項

- リポジトリ直下に Rust workspace を新規作成する
- アプリ用ソース、設定、ドキュメント、テストを新規追加する

## 4. 機能仕様

### 4.1 正常系の仕様

- サーバー起動
  - SQLite データベースに接続する
  - 必要なテーブルがなければ初期化する
  - 少なくとも 1 つの初期サーバーを利用可能にする
  - WebSocket 接続を受け付ける
- サインアップ
  - ユーザー名とパスワードを入力して新規ユーザーを作成する
  - ユーザー名は一意にする
  - パスワードはソルト付きハッシュとして保存する
  - サインアップ成功後、ログイン可能になる
- ログイン
  - ユーザー名とパスワードでログインする
  - 保存済みパスワードハッシュで照合する
  - 成功時はクライアントがチャット画面へ遷移する
- サーバー / チャンネル表示
  - ログイン後、サーバー一覧と選択中サーバーのチャンネル一覧を表示する
  - チャンネルを選択すると、そのチャンネルのメッセージ履歴を表示する
- チャンネル作成
  - 選択中サーバーに新規チャンネルを作成できる
  - 作成後、接続中クライアントのチャンネル一覧に反映される
- チャンネル削除
  - 選択中サーバーのチャンネルを削除できる
  - 削除後、接続中クライアントのチャンネル一覧から消える
  - 削除対象チャンネルを表示中のクライアントは、別チャンネル選択または未選択状態へ遷移する
- メッセージ送信
  - ログイン済みユーザーが選択中チャンネルにテキストメッセージを送信できる
  - サーバーは `message_id`、`user_id`、`channel_id`、`timestamp`、本文を SQLite に保存する
  - 保存成功後、同じチャンネルを見ている接続中クライアントへ配信する
- メッセージ受信
  - クライアントは WebSocket 経由で新着メッセージを受信し、メッセージ一覧に追加表示する
  - クライアント 1 から送ったメッセージがクライアント 2 に表示され、その逆も成立する

### 4.2 異常系の仕様

- サインアップ
  - ユーザー名が空の場合はエラーにする
  - パスワードが空の場合はエラーにする
  - 既存ユーザー名の場合はエラーにする
- ログイン
  - ユーザーが存在しない場合はエラーにする
  - パスワードが一致しない場合はエラーにする
  - エラー時に、パスワードのどの部分が誤っているかは表示しない
- WebSocket
  - 未ログインまたは無効なセッションでの接続は拒否する
  - 不正な JSON または未知のメッセージ種別を受け取った場合は、接続全体を即時クラッシュさせず、エラー応答またはログ記録を行う
- チャンネル作成
  - チャンネル名が空の場合はエラーにする
  - 同一サーバー内で同名チャンネルがある場合はエラーにする
- チャンネル削除
  - 存在しないチャンネルの削除はエラーにする
  - 最後の 1 チャンネルを削除できるかは未確定事項とする
- メッセージ送信
  - チャンネル未選択の場合は送信できない
  - 空メッセージは送信できない
  - 存在しない、または削除済みチャンネルへの送信はエラーにする
- SQLite
  - DB 初期化または書き込みに失敗した場合、サーバーはエラーをログに出し、該当リクエストを失敗扱いにする
  - 起動に必要な DB 接続ができない場合、サーバーは起動失敗として終了してよい

### 4.3 境界条件

- ユーザー名
  - 空文字は禁止
  - 最大長は仮仕様として 32 文字
  - 使用可能文字は未確定。実装時はまず表示崩れを避けるため前後空白を trim する
- パスワード
  - 空文字は禁止
  - 最小長は未確定
  - 最大長は仮仕様として 256 文字
- チャンネル名
  - 空文字は禁止
  - 同一サーバー内で重複禁止
  - 最大長は仮仕様として 64 文字
- メッセージ本文
  - 空文字は禁止
  - 最大長は仮仕様として 2000 文字
- タイムスタンプ
  - サーバー側で生成する
  - UTC の日時として扱う
- ID
  - `user_id`、`server_id`、`channel_id`、`message_id` はサーバー側で生成する
  - 具体的な ID 型は仮仕様として UUID

### 4.4 非機能要件

- パフォーマンス
  - MVP では同一マシン上の少人数利用を想定する
  - 大規模同時接続や大量履歴の最適化は対象外
- セキュリティ
  - パスワード平文保存は禁止
  - 汎用 SHA-256 だけの高速ハッシュは禁止
  - ログにパスワードやパスワードハッシュを出力しない
  - 認証失敗時のエラー表示は詳細にしすぎない
- 保守性
  - `common` crate に通信 DTO と共通 ID 型を集約する
  - サーバーの DB 層、認証層、WebSocket 層を分ける
  - クライアントの UI 状態、通信処理、OS 固有処理を分ける
- 互換性
  - 今回は開発環境での動作を優先する
  - OS 固有 API への直接依存は隔離する
- ログ
  - サーバー起動、接続、認証失敗、DB エラー、WebSocket エラーはログ対象にする
  - 個人情報やパスワードはログに含めない

## 5. UI / API / データ仕様

### 5.1 UI仕様

- ログイン / サインアップ画面
  - サーバー接続先入力
  - ユーザー名入力
  - パスワード入力
  - ログインボタン
  - サインアップボタン
  - エラーメッセージ表示領域
- チャット画面
  - 左側にサーバー一覧
  - その隣に選択中サーバーのチャンネル一覧
  - 中央または右側にメッセージ一覧
  - 下部にメッセージ入力欄と送信ボタン
  - チャンネル作成ボタン
  - チャンネル削除ボタン
- UI 挙動
  - 未ログイン時はチャット画面を表示しない
  - ログイン成功後にサーバー / チャンネル / メッセージを読み込む
  - 通信中や失敗時はユーザーが状況を把握できる表示を出す
  - パスワード入力はマスク表示にする

### 5.2 API仕様

実装方針は REST 風 HTTP API + WebSocket の併用を仮仕様とする。ユーザー回答で API 形状は明示されていないため、後続フェーズで実装計画に落とす際にこの仮仕様を採用するか確認してもよい。

- HTTP API 仮仕様
  - `POST /api/signup`
    - 入力: `username`, `password`
    - 出力: 成功 / 失敗
  - `POST /api/login`
    - 入力: `username`, `password`
    - 出力: セッションまたは認証トークン
  - `GET /api/bootstrap`
    - 入力: 認証情報
    - 出力: サーバー一覧、チャンネル一覧、初期表示用データ
  - `POST /api/servers/{server_id}/channels`
    - 入力: チャンネル名
    - 出力: 作成されたチャンネル
  - `DELETE /api/channels/{channel_id}`
    - 入力: 認証情報
    - 出力: 成功 / 失敗
  - `GET /api/channels/{channel_id}/messages`
    - 入力: 認証情報
    - 出力: メッセージ履歴
- WebSocket 仮仕様
  - `GET /ws`
  - 認証済みクライアントのみ接続可能
  - クライアントからサーバーへのイベント:
    - `send_message`
    - `subscribe_channel`
  - サーバーからクライアントへのイベント:
    - `message_created`
    - `channel_created`
    - `channel_deleted`
    - `error`

### 5.3 データ仕様

- SQLite テーブル仮仕様
  - `users`
    - `id`
    - `username`
    - `password_hash`
    - `created_at`
  - `servers`
    - `id`
    - `name`
    - `created_at`
  - `channels`
    - `id`
    - `server_id`
    - `name`
    - `created_at`
    - `deleted_at` または物理削除
  - `messages`
    - `id`
    - `channel_id`
    - `user_id`
    - `body`
    - `created_at`
  - `sessions` または認証トークン管理相当
    - `token_hash` または `token`
    - `user_id`
    - `created_at`
    - `expires_at`
- 通信 DTO
  - `common` crate に置く
  - `user_id`、`message_id`、`timestamp` を含める
  - サーバーから返すメッセージには送信者表示名を含めてもよい
- 保存方式
  - メッセージ履歴は SQLite に保存する
  - パスワード平文は保存しない

### 5.4 設定仕様

- サーバー設定
  - listen host / port を指定可能にする
  - SQLite DB パスを指定可能にする
  - 未指定時の仮仕様:
    - host: `127.0.0.1`
    - port: `3000`
    - DB path: `./data/chat.sqlite3`
- クライアント設定
  - 接続先 URL を UI から入力できる
  - 未指定時の仮仕様:
    - `http://127.0.0.1:3000`

### 6.1 単体テスト観点

## 9. 後続フェーズへの引き継ぎ

- ブランチ計画では、まず Cargo workspace と `common` crate を基盤として分ける
- 次にサーバーの DB 初期化、ユーザー認証、チャンネル、メッセージ永続化を実装単位に分ける
- WebSocket のリアルタイム配信は、DB 保存と API の基盤ができてから扱う
- クライアントはログイン / サインアップ画面、チャット画面、通信層、状態管理を分ける
- `iced` UI と非同期通信の連携はリスクが高いため、検証を厚くする
- パスワードハッシュと認証トークン管理はセキュリティ上のリスクが高いため、平文保存やログ漏えいがないことを重点確認する
- SQLite 永続化は、サーバー再起動後の履歴確認まで検証する
- 最初の統合目標は「2 クライアントでログインし、同じチャンネルで双方向メッセージ送受信できること」とする
- チャンネル作成・削除はリアルタイム反映まで確認対象にする
- Windows / macOS 対応は今回の動作保証外だが、OS 固有処理の隔離方針はブランチ計画に含める
